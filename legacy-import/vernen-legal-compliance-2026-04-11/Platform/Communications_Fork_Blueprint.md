## What the Communications Fork Is

The Communications Fork is the compliance layer applied to the channel itself — not what is communicated, but whether the communication infrastructure is what it claims to be.

Every other fork — Visual, Audio, Software, CITIZEN — operates on a device that connects to a network through a tower. If that device is cloned, if that tower is fake, if that SIM has been swapped, the entire trust chain of every other fork collapses. A Visual fork document audit means nothing if the device that captured it is not what it says it is. An Audio fork translation record means nothing if the phone that recorded it was compromised at the carrier level.

The Communications Fork is the foundation that every other fork rests on.

**The core principle:** before any data, evidence, communication, or compliance finding leaves a device, that device must prove it is what it claims to be — and the network must prove the same.

## The Problem — Documented and Current

| Threat | Reality |
|---|---|
| **SIM swap fraud** | FBI IC3: $68M+ confirmed losses in 2021 alone — the floor, not the ceiling |
| **IMSI catchers / Stingrays** | DHS confirmed unauthorized cell sites operating in Washington DC in 2018. No public attribution to this day. |
| **SS7 protocol exploitation** | Every 2G/3G handoff rides a protocol designed in 1975 with zero authentication. Exploitable by any state or non-state actor with $50K in equipment. |
| **Diameter protocol (4G/5G)** | SS7's replacement. Better, but still exploitable. Documented attacks in academic literature and in the wild. |
| **Rogue eNodeBs / fake base stations** | Commercial hardware that impersonates a carrier tower: under $5,000. Deployed against journalists, attorneys, activists, domestic violence survivors. |
| **Cloned handsets** | Documented in trafficking and stalking cases. Device identity spoofed at the hardware layer. |
| **SUCI / 3GPP mitigations** | Encrypt the IMSI over the air interface. Partial mitigation only — assumes the network itself is honest. The tower can still be fake. |

The fundamental architecture of cellular communications still assumes three things that are no longer safe to assume:
1. The device is what it claims to be
2. The network is what it claims to be
3. The tower in between is operated by the carrier it says it is

None of these assumptions hold in 2026.

## The Architecture

### Device Layer — Compliance Citizen in the TEE

Every modern phone has a **Trusted Execution Environment (TEE)**:
- ARM TrustZone (Android devices)
- Secure Enclave (Apple devices)
- TPM chips (Windows/Linux devices)

These are hardware-isolated execution environments that even a compromised operating system cannot access. They are already used for:
- Biometric authentication storage
- Payment credential isolation (Apple Pay, Google Pay)
- Device encryption key management

The **Vernen Communications Compliance Citizen** runs inside this TEE. It:
- Holds a device-specific cryptographic identity issued at enrollment
- Generates attestation signatures before every network registration attempt
- Verifies the network's corresponding attestation before accepting the connection
- Logs every connection event to the device's local tamper-evident ledger
- Refuses to connect if attestation fails — the device simply does not transmit

This is not software that can be bypassed by a compromised OS. It runs below the OS in the hardware trust boundary.

### Network Layer — Vernen Compliance Gateway

A **Vernen Compliance Gateway** sits in front of the carrier core network (between the radio access network and the core). Before a device is allowed to register:

1. Device sends attestation signature (generated by TEE Citizen)
2. Gateway verifies signature against the Vernen federated ledger
3. Gateway sends its own attestation signature back to the device
4. Device TEE Citizen verifies the network is a known, certified, uncompromised endpoint
5. If both sides pass: connection established, event logged
6. If either side fails: connection refused, attempt logged as anomaly

A fake tower cannot pass step 3. A cloned device cannot pass step 1. A SIM-swapped device cannot pass step 1 because the attestation is tied to the hardware TEE, not the SIM.

### The Verification Ledger — Certificate Transparency Architecture

The ledger that records all attestation events uses a **Certificate-Transparency-style architecture**:

```
[Attestation Event]
       │
       ▼
[Merkle Tree Log — append-only, tamper-evident]
       │
       ├── Vernen log operator
       ├── FCC log operator
       ├── EFF log operator (civil liberties check)
       ├── Academic consortium log operator
       └── Carrier log operators (AT&T, Verizon, T-Mobile)
       │
       ▼
[Public auditability — anyone can verify any attestation]
```

No single operator — including Vernen — can alter the log. An entry exists when it appears in at least two independent logs. This is identical to how Certificate Transparency works for HTTPS certificates today: browsers reject any HTTPS cert that doesn't appear in at least two independent CT logs.

**What is logged:** attestation signature · timestamp · device class (not identity) · pass/fail
**What is NOT logged:** who the device belongs to · who it called · message content · location · any data protected by *Carpenter v. United States* (2018)

The architecture literally cannot capture connection metadata. It is designed this way deliberately.

### eSIM Integration — Existing Leverage Point

The GSMA SGP.22 standard for remote eSIM provisioning already provides:
- Cryptographic device identity at the SIM layer
- Remote attestation capability
- Standardized provisioning protocol

The Vernen Communications Citizen builds on this existing standard rather than replacing it. The TEE Citizen wraps the eSIM attestation with an additional compliance layer — the eSIM proves device identity, the TEE Citizen proves the compliance module is running and unmodified.

## The FCC Mandate Path

The FCC already has the authority. The precedents exist:

| Precedent | What it established |
|---|---|
| **E911** (1996–2010) | FCC mandated location accuracy on every handset sold in the US |
| **STIR/SHAKEN** (2021) | FCC mandated caller ID authentication across all US carriers |
| **CALEA** (1994) | Congress mandated telco lawful-intercept capability |
| **HAC** (Hearing Aid Compatibility) | Every handset sold US-side must comply |
| **Equipment Authorization Program** | Every radio device sold in US must be FCC-certified before shipping |

A rule requiring device attestation for US carrier network registration fits squarely inside Title III authority the FCC has exercised for fifty years.

### Deployment Timeline (Realistic)

```
Year 1 — Voluntary Certification
Vernen Communications Compliance Certification available.
Carriers and device manufacturers opt in voluntarily.
Like FIDO certification or UL listing — no mandate yet.

STIR/SHAKEN took this exact path.

US profile integrated into global cellular standard.
Roaming compatibility restored.
GSMA alignment.
```

## 4th Amendment Design Constraints

*Carpenter v. United States* (2018) established that cell-site location information — even metadata about which towers a device connected to — is protected by the 4th Amendment and requires a warrant.

The Communications Fork is designed so that Vernen **cannot** produce this information even under subpoena:

- No connection metadata is stored
- No device identity is logged (only device class — "Android TEE certified" not "Michael's phone")
- No location data enters the system
- No call records, message content, or communication metadata is captured
- The only data that exists is: an attestation signature, a timestamp, and a pass/fail

This is not a policy choice that can be reversed. It is an architectural constraint. The system is incapable of the surveillance it might otherwise enable.

This is the answer to the civil liberties objection before it is raised.
